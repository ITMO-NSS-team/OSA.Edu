# OSA.Edu

Веб-система для автоматизированной предварительной проверки выпускных квалификационных работ и диссертаций.

OSA.Edu анализирует структуру, содержание и формальное оформление документа, находит потенциальные нарушения, приводит доказательства из текста и формирует понятный отчёт для последующей экспертной проверки.

Система не заменяет научного руководителя, нормоконтролёра или диссертационный совет. Итоговые результаты являются рекомендациями и сигналами для ручной проверки.

## Основная идея

Проект использует гибридный подход:

- точные и формальные требования проверяются программным кодом;
- содержательные требования анализируются языковой моделью;
- каждое замечание связывается с конкретной страницей, блоком и цитатой;
- пользователь вручную подтверждает структуру документа до запуска основных правил.

Главный приоритет проекта — содержательная проверка:

- научная новизна;
- логика работы;
- соответствие цели и задач;
- связь положений с главами;
- анализ аналогов и прототипов;
- доказательность результатов;
- эксперименты;
- согласованность выводов и заключения.

Орфографические и мелкие оформительские замечания имеют более низкий приоритет и не должны доминировать в итоговой оценке.

## Возможности

### Поддержка документов

- PDF с текстовым слоем;
- DOCX;
- работы большого объёма;
- сохранение страниц и стабильных идентификаторов текстовых блоков.

### Построение структуры

Система автоматически выделяет:

- название работы;
- реферат;
- введение;
- цель;
- задачи;
- положения, выносимые на защиту;
- главы и подразделы;
- выводы по главам;
- заключение;
- список литературы;
- приложения.

После построения карты проверка останавливается. Пользователь может исправить названия, типы и границы диапазонов, добавить пропущенные элементы и только после этого подтвердить структуру.

### Профили правил

В проекте предусмотрены два профиля:

- **основной профиль** — 65 правил;
- **полный профиль** — 227 правил.

Правила маршрутизируются только в те области документа, где они должны применяться. Например, требования к названию не проверяются в библиографии, а требования к выводам не анализируют весь документ целиком.

### Детерминированные проверки

Без LLM проверяются условия, которые можно установить точно:

- количество слов в названии;
- регистр и знаки препинания в нумерованных пунктах;
- оформление подписей рисунков и таблиц;
- ссылки на рисунки, таблицы и источники;
- диапазоны библиографических ссылок;
- запрещённые слова и конструкции;
- использование личных местоимений;
- сокращения и первое употребление терминов;
- оформление библиографии;
- наличие обязательных разделов;
- нумерация и структура выводов.

### Содержательные проверки

Языковая модель используется для требований, где требуется понимание смысла:

- корректность научной новизны;
- связь цели и задач;
- соответствие положений главам;
- наличие аналогов и прототипа;
- недостатки прототипа и способ их устранения;
- достаточность экспериментального подтверждения;
- соответствие результатов выводам;
- согласованность глав, заключения и заявленных результатов;
- наличие содержательных противоречий.


### Проверка доказательств

Для подтверждённого нарушения модель должна вернуть:

- существующий `blockId`;
- точную цитату;
- страницу;
- объяснение;
- исправление.

Приложение проверяет, что цитата действительно содержится в указанном блоке и относится к разрешённой области правила. Неподтверждённое замечание не принимается как уверенное нарушение.

### Контроль противоречий

После завершения всех правил система проверяет результаты на логическую согласованность.

Пример:

```text
CORE-2-3: прототип не выбран
CORE-2-1: отличие от прототипа полностью доказано
```

Такой уверенный `pass` понижается до `uncertain`, поскольку выводы противоречат друг другу.

### Дедупликация замечаний

Несколько правил могут описывать одну физическую проблему. Например, точка после подписи может одновременно относиться к нескольким требованиям.

В отчёте все связанные правила сохраняются, но одно замечание не должно несколько раз снижать итоговую оценку.

### Отчёт

Итоговый протокол содержит:

- предварительную оценку;
- покрытие правил;
- нарушения;
- выполненные требования;
- неопределённые результаты;
- неприменимые и необработанные правила;
- страницы и цитаты;
- рекомендации по исправлению;
- матрицы полного покрытия;
- связанные правила;
- техническую информацию о запросах OpenRouter.

Поддерживаемые статусы:

```text
pass            требование выполнено
violation       нарушение подтверждено
uncertain       доказательств недостаточно
not_checked     проверка не выполнена
not_applicable  правило неприменимо
```

## Архитектура

```text
PDF / DOCX
    ↓
извлечение текста и страниц
    ↓
разбиение на блоки с blockId
    ↓
построение смысловой карты документа через LLM
    ↓
ручная проверка и подтверждение структуры
    ↓
маршрутизация правил по нужным диапазонам
    ↓
детерминированные и структурные проверки
    ↓
групповые содержательные проверки через OpenRouter
    ↓
валидация цитат и матриц полного покрытия
    ↓
контроль противоречий и дедупликация
    ↓
итоговый отчёт
```



### Основные компоненты

```text
src/
  components/              страницы и компоненты React-интерфейса
  api.ts                   клиент серверного API
  types.ts                 типы фронтенда

server/
  document/                анализ документа и построение карты
  checking/                детерминированные и структурные проверки
  routing/                 маршрутизация правил по фрагментам
  orchestration/           запуск и объединение проверок
  rules/                   загрузка и регистрация правил
  llm.ts                   клиент OpenRouter, retry и диагностика
  queue.ts                 состояния задач и частичный повтор
  report.ts                формирование протокола

config/
  document-map-prompt.txt  промпт построения структуры
  semantic-prompt.txt      промпт содержательной проверки
  rule-routing.json        карта маршрутизации правил

rules-data/
  core.csv                 основной профиль
  soft.csv                 дополнительные правила
```

## Установка

Требования:

- Node.js 20.19 или новее;
- npm;
- API-ключ OpenRouter.

```bash
npm install
npm ci
cp .env.example .env
```

Добавьте ключ в `.env`:

```env
OPENROUTER_API_KEY=sk-or-...
```

Запуск режима разработки:

```bash
npm run dev
```

После запуска:

- интерфейс: `http://127.0.0.1:5173`;
- API: `http://127.0.0.1:8787`.

Отдельный запуск backend:

```bash
npm run dev:server
```

Отдельный запуск frontend:

```bash
npm run dev:web
```

Production-сборка:

```bash
npm run build
npm start
```

Основные переменные среды:

```env

OPENROUTER_DATA_COLLECTION=allow

OPENROUTER_REQUIRE_PARAMETERS=false

OPENROUTER_MAX_CONCURRENT_REQUESTS=1
OPENROUTER_MIN_REQUEST_INTERVAL_MS=3400
OPENROUTER_MAX_ATTEMPTS=4
OPENROUTER_MAX_COMPLETION_TOKENS=16000

LLM_REQUEST_TIMEOUT_MS=300000
MAX_FILE_SIZE_MB=35
```


## Как пользоваться

1. Запустите приложение.
2. Выберите профиль правил.
3. Выберите модель OpenRouter.
4. Загрузите PDF или DOCX.
5. Дождитесь построения структуры.
6. Проверьте найденные диапазоны вручную.
7. Исправьте связи и границы при необходимости.
8. Подтвердите структуру.
9. Запустите проверку правил.
10. Просмотрите доказательства и скачайте протокол.

При сбое отдельных LLM-запросов можно использовать кнопку **«Повторить неудачные LLM-проверки»**. Успешные результаты при этом не пересчитываются.

## Ограничения

- PDF должен содержать текстовый слой; OCR не встроен.
- Извлечение PDF может разрывать слова и смешивать формулы с обычным текстом.
- Шрифты, поля, точное выравнивание и визуальное положение объектов нельзя надёжно проверить только по текстовому слою.
- Связь положений с главами может требовать ручного исправления.
- Содержательные результаты могут различаться между моделями и upstream-провайдерами.
- Числовая оценка является ориентиром по выбранному профилю правил, а не объективной оценкой научной работы.

## Дисклеймер

OSA.Edu предоставляет рекомендации для последующей экспертной проверки. Автоматический отчёт не является официальным заключением, не определяет научную ценность работы и не заменяет решение научного руководителя, нормоконтролёра, рецензента или аттестационной комиссии.
